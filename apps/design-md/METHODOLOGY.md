# Design MD 提取方法论 V1

这份文档是当前项目的第一版提取总表。

它的作用不是对外展示，而是作为我们后续继续处理更多 `design MD` 文件时的统一工作依据。

这不是最终定稿。
后续随着新品牌、新结构、新问题出现，这份文档会继续迭代。

## 目标

我们的目标不是把 `design MD` 变成一个更漂亮的 Markdown 阅读器，而是把不同品牌的设计规范提炼成一个：

- 可比较
- 可判断
- 可复制
- 可继续扩展

的设计知识系统。

## 总原则

1. 先提系统，不先提截图感受。
2. 只提文档里明确出现、能被证据支持的内容。
3. 不为了完整而脑补缺失规范。
4. 数据可以完整，展示必须克制。
5. 中文优先，英文只保留必要术语、变量名、色值、字体名、token 名。
6. 同一主题下，所有品牌尽量用同一套比较框架。
7. 前端展示服务于快速判断和复制，不服务于装饰。

## 统一工作流

1. 先完整阅读文档，判断其重点强调了哪些设计主题。
2. 按现有主题框架拆分内容。
3. 每个主题提成结构化数据，而不是只保留自然语言。
4. 再决定前端如何展示：
   - 是否需要真实可视化
   - 是否支持点击复制
   - 是否只保留简洁示意
5. 对不明确的内容保守处理，不扩写。
6. 如果新品牌出现现有主题无法容纳的新类型，再扩展主题。

## 当前主题框架

1. 网格与容器
2. 颜色系统
3. 字体系统
4. 卡片
5. 层级与阴影
6. 按钮
7. 圆角尺度
8. 间距系统
9. 表单元素
10. 规范边界

## 1. 网格与容器

### 提取内容

- 顶部结构
- 主体容器
- section 组织方式
- 最大宽度或居中方式
- 响应式断点
- 列数变化
- 页面骨架逻辑

### 提取重点

不是只找“几列网格”，而是判断页面是如何组织空间的。

例如：

- Airbnb 更偏响应式 listing grid
- Apple 更偏全屏 section 和居中内容块
- Figma 更偏展陈式 gallery 容器

### 展示原则

- 需要有简化骨架图或结构示意
- 不只放一句说明文字
- 重点是页面骨架，而不是组件细节

### 判断标准

- 有没有抓到主骨架
- 有没有说清 full-width、centered block、grid 的关系
- 有没有提到断点和列数变化

## 2. 颜色系统

### 提取内容

- 名称
- 色值：HEX / RGB / RGBA / gradient
- 用途
- 所属分组
- 原文相关描述

### 必须分组

- 品牌色
- 中性色 / 文字色
- 背景 / surface
- 边框 / 分割线
- 功能色 / 状态色
- 渐变 / 玻璃态色

### 展示原则

- 先看真实颜色，再看文字说明
- 卡片统一尺寸
- 不省略颜色
- 颜色少也不能自动拉宽卡片
- 支持点击复制

### 复制原则

优先复制：

- 原文相关片段
- 或结构化值：名称 + 数值 + 用途

### 判断标准

- 有没有漏色
- 用途是否正确
- 分组是否符合原文
- 前端颜色是否真实渲染

## 3. 字体系统

### 提取内容

- 字体家族
- 字号
- 字重
- 行高
- 字距
- 大小写规则
- 示例文本
- 使用场景

### 展示原则

- 必须真实渲染排版
- 不能只列参数
- 英文品牌优先用英文样例
- 重点是让人看到层级差异

### 判断标准

- 展示出来的排版是否真实反映参数
- 是否覆盖标题、正文、标签、微文案等主要层级

## 4. 卡片

### 提取内容

- 卡片类型
- 背景
- 圆角
- 阴影 / 边框
- 图片和文字关系
- 内部内容层级
- 使用场景

### 核心原则

不能照着截图“模仿一个卡片”，而要提取文档里卡片被规定了什么。

### 展示原则

- 让用户一眼知道适不适合自己
- 展示卡片的必要信息
- 避免拍脑袋补内容

### 判断标准

- 结构是否来自文档
- 是否展示了卡片真正重要的信息
- 是否避免了随意造型

## 5. 层级与阴影

### 提取内容

- 阴影
- hover 抬升
- focus ring
- dashed outline
- glass / blur
- background contrast
- 哪些地方故意 flat

### 核心原则

深度不等于 box-shadow。

要先判断这个品牌真正靠什么制造层级。

### 当前理解

- Airbnb：三层暖阴影 + hover shadow + active/focus ring
- Apple：glass nav + 场景对比 + 少量柔和阴影
- Figma：surface 对比 + 少量阴影 + dashed focus

### 展示原则

目前前端采用的是简单直接卡片展示。

也就是说：

- 方法论上要知道“层级来源”和“阴影本身”是两回事
- 但展示上可以按你偏好的直接卡片方式呈现

### 判断标准

- 是否抓到真正深度来源
- 是否漏掉了 ring / blur / contrast
- 是否过度脑补阴影系统

## 6. 按钮

### 提取内容

- 主 CTA
- 次级按钮
- 链接型 CTA
- tab / chip / pill
- 图标 / 圆形按钮
- 背景、文字、边框
- 圆角
- padding
- 字体
- 状态：default / hover / active / focus
- 用途

### 核心原则

按钮提取不是摆几个按钮，而是提按钮家族。

### 展示原则

- 按角色展示
- 每个品牌展示同类角色，方便横向比较
- 优先保留按钮本体，不保留多余解释
- 去掉重复标题、重复说明、重复外框

### 判断标准

- 是否提全主要角色
- 是否把链接误当按钮
- 是否把文档未说的状态补了出来

## 7. 圆角尺度

### 提取内容

- 关键圆角节点
- 对应控件家族：
  - 链接
  - 按钮
  - 卡片
  - 容器
  - pill
  - circle
- 签名圆角值

### 核心原则

圆角不是连续大小表，而是一组几何家族。

### 展示原则

- 去掉重复展示
- 中文化用途说明
- 特殊形状要适配排版
- pill 和 circle 不能套普通方块布局

### 排版原则

- 图形列用自适应宽度
- 文案列单独占剩余空间
- 防止胶囊形状压住文字

### 判断标准

- 是否抓到签名圆角
- 是否区分矩形、pill、circle
- 是否避免重复信息

## 8. 间距系统

### 提取内容

- spacing scale
- 特殊值
- 节奏特征

### 内部理解

虽然前端展示很简，但内部需要区分：

- 微调刻度
- 常用 UI 刻度
- 内容 / section 留白

### 展示原则

- 展示极度克制
- 只保留一条间距示意尺
- 不加多余解释卡片

### 判断标准

- 是否提到基础单位
- 是否保留特殊值
- 是否能从这条尺度中看出节奏差异

## 9. 表单元素

### 提取内容

- 搜索
- 输入框
- textarea
- filter / chips
- 焦点态
- 边框 / 背景 / 圆角
- 状态和用途

### 核心原则

不强行套用“通用表单库”模板。

先判断这个品牌真正重视的输入方式是什么。

### 当前理解

- Airbnb：重点是搜索入口和筛选 pills
- Apple：重点是 search / filter / 精密 focus
- Figma：重点是黑白输入容器和 dashed focus

### 展示原则

- 至少能看出默认态和焦点态
- 如果搜索和筛选属于同一系统，要一起展示
- 不补不存在的 error / validation 规范

### 判断标准

- 是否抓到品牌真正重视的输入场景
- 是否乱补表单状态

## 10. 规范边界

### 提取内容

- 要做
- 不要做
- 不可破坏的设计原则
- 明确禁止的用法

### 展示原则

- 直接
- 短句
- 对照清晰

### 判断标准

- 是否提炼出真正可执行的边界
- 是否避免空泛口号

## 前端展示总原则

1. 比较模式优先。
2. 同一主题下，品牌纵向排列，方便滚动比较。
3. 左侧品牌列表负责快速跳转。
4. 视觉内容优先于解释文字。
5. 去掉重复信息。
6. 中文优先，英文只保留必要术语。
7. 一切以快速判断和方便复制给 AI 为目标。

## 复制总原则

复制不一定只复制 CSS。

更适合复制的是：

- 原文相关片段
- 参数 + 用途
- 结构化描述

因为使用对象很多时候是 AI，而不是人类读者。

## 新品牌接入规则

1. 先按现有 10 个主题扫描。
2. 能归类的先归类。
3. 不能归类的，判断是否真的是新主题。
4. 信息明确的做真实可视化。
5. 信息不明确的做保守展示。
6. 不为了页面完整强行填满每个主题。

## 版本说明

这是 V1。

它的特点是：

- 已经足够支撑继续提取更多品牌
- 但仍然会随着新品牌和新问题继续迭代
- 页面展示策略也会跟着一起修正

后续如有更新，建议直接在这份文档上继续追加版本记录。
